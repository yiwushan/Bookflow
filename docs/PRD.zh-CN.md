# 《书流》(BookFlow) 产品需求文档（PRD）

## 0. 文档信息
- 版本：`v1.1 (Section-First Draft)`
- 日期：`2026-03-28`
- 状态：`可进入技术评审`
- 产品定位：将个人吃灰电子书库，转化为高成瘾性“小红书式”碎片化知识信息流。
- 核心策略：用短平快滑动交互降低长篇阅读启动阻力，并通过“回看上文 -> 连续深读”机制触发主动读书。

---

## 1. 产品目标

### 1.1 业务目标
1. 让用户从“囤书不读”转为“日常轻阅读”。
2. 在不牺牲知识质量的前提下，提升阅读频次和留存。
3. 让单本书形成“可收集、可解锁、可回看”的连续体验。

### 1.2 北极星指标（North Star）
- 人均每日“完整小节完成数”（`section_complete`）。

### 1.3 核心结果指标（MVP）
- `D1` 留存 >= 35%
- `D7` 留存 >= 18%
- 人均每日完整小节完成数 >= 2
- 连续阅读深度（同一本书连续阅读条数）P50 >= 3
- Feed 到“进入上下文轴”转化率 >= 20%
- “书籍主页”访问转化率（从 Feed 进入书主页）>= 15%

### 1.4 约束指标（Guardrail）
- AI 改写后“事实偏移率”< 2%
- 用户对“标题党”负反馈率（快速划走+点踩）< 10%
- 单切片平均生成成本控制在设定预算内（本地模型优先）
- 技术内容（公式/定理/代码）结构保真率 >= 99%
- 过度碎片化率（进入后快速退出且不回看上文）< 20%

---

## 2. 目标用户与场景

### 2.1 目标用户
1. 有大量 PDF/EPUB 存量但阅读断续的知识型用户。
2. 通勤/排队等碎片时间读者。
3. 希望“轻量获取 + 深入追读”两种模式并存的人群。

### 2.2 核心使用场景
1. 用户导入书库后，系统自动拆书并生成可滑动阅读 Feed。
2. 用户刷到感兴趣片段，向左滑进入上下文，连续阅读本书。
3. 用户在书主页“点亮”切片，形成收集和进度反馈。
4. 用户被“那年今日”回忆帖召回，完成复习与二次阅读。

---

## 3. 范围定义

### 3.1 MVP（必须）
1. PDF/EPUB 解析、清洗、切片、标签化离线管道。
2. 切片智能压缩与“诱饵”标题生成（按需触发）。
3. 首页伪推荐 Feed（单列优先）、上下滑切换、左滑进入阅读模式。
4. 书籍主页拼图式切片墙（灰/亮状态）。
5. 点赞、评论、阅读时长与完成行为埋点。
6. 每日“一个月前/一年前”回忆帖插入 Feed。
7. 技术书保真模式（公式/代码块不改写，仅做释义层）。
8. 混合渲染策略（文本重排优先，PDF 裁切兜底）。

### 3.2 非 MVP（后续）
1. 用户间关注关系与社交分发。
2. 多端实时同步与云端协作笔记。
3. 复杂多目标学习排序模型（先规则+轻模型）。

## 4. 功能需求（按模块）

## 模块一：离线前处理管道（数据准备层）

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-101 | 支持导入 `PDF/EPUB` 文件，建立书籍元数据 | P0 | 成功入库率 >= 98%，失败可重试并记录日志 |
| FR-102 | 文本清洗：去乱码、页眉页脚、多余换行、异常空白 | P0 | 清洗后文本可连续阅读，异常字符占比 < 0.5% |
| FR-103 | 以“完整小节”优先切片，过长再做二次分段 | P0 | 80% 切片与原书小节边界对齐；切片边界不截断关键推导 |
| FR-104 | 为切片生成 2-3 个标签（本地轻模型优先） | P0 | 每切片有标签且语义相关；标签空值率 < 1% |
| FR-105 | 质量评分与回退机制（低质切片重新切分） | P1 | 低质量切片可自动重试，重试后通过率 >= 85% |
| FR-106 | 技术结构检测：公式、图表、代码块、定理环境识别 | P0 | 技术书切片中结构识别召回率 >= 95% |
| FR-107 | 渲染模式判定：重排文本 or 原页裁切 | P0 | 每切片有渲染模式，判定可追溯 |

补充规则：
1. 切片目标时长：建议 `3-8` 分钟阅读量（优先按小节完整性）。
2. 目录缺失时，退化为语义段落 + 字数阈值切分。
3. 处理产物需支持幂等：重复运行不会产生重复切片。
4. 检测到技术内容时优先按“概念/定理/例题边界”切片，避免截断推导链。
5. 单小节过长时按“定义->推导->例题”拆成多段，但保留连续链路。

## 模块二：内容生成引擎（运行时服务层）

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-201 | 超长或过干切片触发压缩/重写（仅叙述性文本） | P0 | 完整小节完成率提升（A/B） |
| FR-202 | 生成“诱饵文案”：问题导向钩子优先 | P0 | 每切片产出 1 条；人工抽检可读率 >= 90% |
| FR-203 | 非首切片生成“AI 前情提要” | P0 | 前情提要长度 30-60 字，语义连贯无剧透过量 |
| FR-204 | 生成内容缓存与版本化 | P1 | 同一模型版本下重复请求命中缓存 >= 80% |
| FR-205 | 技术书“保真改写”策略：原文保留+术语解释，不改公式与代码 | P0 | 关键符号、公式、代码段一致率 >= 99% |
| FR-206 | 技术切片可选“双层展示”：原文层 + 通俗解释层 | P1 | 用户可一键切换，解释层可关闭 |

内容安全与质量要求：
1. 不得引入原文没有的关键事实结论（除明确标注推断）。
2. 文风优先“轻、准、短”，避免强营销语气。
3. 所有 AI 生成字段保留 `model_name` 与 `model_version` 便于追溯。
4. 对技术书默认禁用“强悬念标题”，优先“问题导向标题”（如“为什么梯度会消失？”）。
5. 重点目标不是“拖长停留”，而是“引导回看与继续读下一段”。

1. 首页先做单列，双列瀑布流作为实验开关。

3. 技术书默认以“单列沉浸”呈现，不建议双列瀑布混排。
4. Feed 中只展示“小节入口卡”，真正阅读在上下文轴连续展开。

## 5. 信息架构与核心流程

### 5.1 离线流程（批处理）
1. 文件导入 -> 解析提取 -> 文本清洗。
2. 清洗文本 -> 小节识别 -> 小节优先切片 -> 质量评分。
3. 切片 -> 渲染模式判定（重排/裁切）-> 标签生成 -> 入库（`chunks + tags`）。
4. 可选预生成：诱饵标题、前情提要（节省线上时延）。

### 5.2 在线流程（请求时）
1. 用户请求 Feed。
2. 排序服务按“连续性 + 标签匹配 + 新鲜度 + 探索度 + 去重”返回候选切片。
3. 缓存命中则直接返回；未命中再调用内容生成引擎。
4. 用户交互行为实时上报，异步更新偏好与进度。

### 5.3 技术书专用流程
1. 识别 `book_type=technical` 后启用保真链路。
2. 先返回原文切片（公式/代码原样），再异步补充释义层。
3. 推荐优先“同章连续 + 前置知识已读”而非随机探索。
4. 对“例题/证明”切片提升上下文加载权重，减少断裂感。

### 5.4 渲染决策流程（重排 vs PDF 裁切）
1. 默认 `reflow`：段落文本、普通叙述、可稳定抽取的标题与列表。
2. 触发 `crop`：复杂公式、密集表格、代码排版敏感、抽取置信度低。
3. `crop` 卡片保留原页锚点（`page + bbox`），支持一键查看前后页。
4. 每张卡片都带“查看原文”入口，避免因重排造成理解偏差。

### 5.5 回忆帖流程（定时任务）
1. 每日巡检 `T-30天` 与 `T-365天` 的高价值互动切片。
2. 生成回忆包装文案（可选）并写入 `memory_posts`。
3. 次日 Feed 按比例插入（建议 5%-10%）。

## 6. 数据模型（MVP）

### 6.1 核心表
1. `books`：书元信息与处理状态。
2. `book_chunks`：切片正文、摘要、诱饵文案、前情提要。
3. `tags` / `chunk_tags`：标签与映射关系。
4. `users`：用户基础信息。
5. `user_tag_profile`：用户标签偏好向量（可增量更新）。
6. `interactions`：曝光、进入上下文轴、回看、完成、点赞、评论、滑走事件。
7. `reading_progress`：按书聚合的阅读进度。
8. `memory_posts`：回忆帖任务与投放状态。

### 6.2 关键字段建议
- `book_chunks.read_state`：`unread / reading / done`
- `interactions.event_type`：`impression / enter_context / backtrack / section_complete / like / comment / skip / confusion`
- `book_chunks.content_version`：用于追踪 AI 改写版本
- `memory_posts.source_date`：原始互动发生日期（支持月前/年前筛选）
- `books.book_type`：`general / fiction / technical`
- `book_chunks.has_formula` / `has_code` / `has_figure`
- `book_chunks.fidelity_score`：改写保真评分
- `book_chunks.prerequisite_chunk_ids`：前置知识切片集合
- `book_chunks.render_mode`：`reflow / crop`
- `book_chunks.source_anchor`：原文定位（页码与区域）

## 7. 推荐与排序策略（MVP 规则版）

### 7.1 候选召回
1. 基于用户高权重标签召回。
2. 加入探索池（低权重新标签）占比 20%。
3. 加入回忆池占比 5%-10%。

### 7.2 排序打分（示例）
`score = 0.30*context_continuity + 0.25*tag_match + 0.15*next_section_prob + 0.10*novelty + 0.10*recency + 0.10*memory_bonus - fatigue_penalty`

技术书模式加权（覆盖默认公式）：
`score_tech = 0.35*concept_continuity + 0.20*prerequisite_ready + 0.20*next_section_prob + 0.15*tag_match + 0.10*recency - fatigue_penalty`

### 7.3 去重与节流
1. 同一本书最多连续出现 2 条。
2. 已明确点踩标签 24 小时内降权。
3. 同一切片 48 小时内不过度重复曝光。
4. 技术书连续深读模式下允许同书连续出现，默认上限放宽至 5 条。

## 8. 接口草案（MVP）

1. `POST /v1/library/import`：导入书籍并触发离线处理。
2. `GET /v1/feed`：拉取信息流（支持 `cursor`）。
3. `GET /v1/chunks/{chunk_id}`：获取切片详情。
4. `GET /v1/chunks/{chunk_id}/context`：获取前后切片。
5. `GET /v1/books/{book_id}`：书主页信息与切片状态。
6. `POST /v1/interactions`：上报曝光/进入上下文轴/回看/小节完成/滑走等行为。
7. `POST /v1/chunks/{chunk_id}/like`：点赞。
8. `POST /v1/chunks/{chunk_id}/comments`：评论。
9. `GET /v1/memory/today`：当日可插入回忆帖。

---

## 9. 非功能需求

1. Feed 接口 `P95 < 300ms`（命中缓存）。
2. 内容生成失败要可降级：展示原始清洗文本。
3. 离线处理任务支持断点续跑与失败重试。
4. 埋点链路具备幂等与去重机制，防重复累计进度。
5. 支持本地模型与低价 API 的可配置切换。
6. 技术内容渲染正确率（公式/代码）>= 99%。
7. `crop` 模式卡片首屏加载 `P95 < 500ms`。

## 10. 实验与验收

### 10.1 A/B 实验优先级
1. 诱饵文案风格：问题导向句 vs 金句。
2. 切片粒度：完整小节 vs 小节内二段拆分。
3. Feed 形态：单列 vs 双列。

### 10.2 MVP 验收口径
1. 完成端到端闭环：导入一本书 -> 产出切片 -> Feed 展示 -> 互动回流 -> 进度更新。
2. 关键事件埋点完整并可回放用户路径。
3. 回忆帖任务可稳定运行 7 天无中断。

---

## 11. 里程碑建议（4 个迭代）

1. Sprint 1：离线管道（解析/清洗/切片/标签）与数据库落地。
2. Sprint 2：Feed + 切片渲染 + 基础交互 + 埋点。
3. Sprint 3：内容生成引擎（压缩、诱饵、前情提要）与缓存。
4. Sprint 4：书主页拼图、回忆帖、排序优化与 A/B 框架。

---

## 12. 风险与应对

1. 文本解析质量波动：增加文档类型识别与多策略清洗模板。
2. AI 改写失真：引入对照抽检与事实一致性评估。
3. 推荐早期冷启动：默认“通用高质量标签池 + 探索配比”。
4. 成瘾设计与伦理平衡：增加“今日完整小节提醒”与可关闭回忆帖。
5. 技术书被过度碎片化：加入“学习模式”与前置知识链校验，优先连续阅读。
6. 公式/代码 OCR 错误：保留原文片段索引，失败时回退到原页截图锚点。
7. 单用户私有化可维护性：优先本地优雅降级，避免过重微服务拆分。

## 14. 连续协作模式（防中断续作）

目标：当任一协作者（人或 AI）中断时，下一位可在 5 分钟内无缝接手。

固定入口：
1. `docs/START_HERE.md`
2. `docs/STATE.json`
3. `docs/TASK_BOARD.md`
4. `docs/specs/v0_scope.md`
5. `docs/FIRST_PRODUCT.md`

执行规则：
1. 优先执行 `TASK_BOARD.md` 的 `NOW` 任务，不擅自重排。
2. 每完成一个 `NOW`，最小更新 `STATE.json` 与 `TASK_BOARD.md`。
3. `SESSION_LOG.md` 仅在里程碑结束或出现阻塞时写短条目。
4. `DECISIONS.md` 仅记录高风险/不可逆取舍（接口、数据模型、产品方向）。
5. 默认测试门禁采用“smoke + 改动模块相关测试”，全量测试仅在里程碑、数据模型/API 变更或回归信号出现时触发。
